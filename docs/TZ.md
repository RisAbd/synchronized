# ТЗ — synchronized

Статус: черновик. Цель документа — разбить крупную идею на **независимые модули** с чёткими
контрактами (JSON), чтобы каждую часть можно было делать и тестировать отдельно.

---

## 1. Обзор архитектуры

Данные текут по цепочке, каждая стрелка — файл/структура (контракт), а не жёсткая связь:

```
источник ──▶ [ingest] ──▶ audio.wav ──▶ [asr] ──▶ transcript.json ──▶ [align] ──▶ sync-map.json
  (url/файл)                (16k mono)            (слова+тайминги)      (ядро)      (тайминг→место)
                                                                                        │
                                       quran.db ──▶ [quran] ──────────────────────────┘
                                    (канон. текст)   (нормализация/индекс)              │
                                                                                        ▼
                                                                  audio + sync-map ──▶ [player]
                                                                                    (крупный текст,
                                                                                     авто-прокрутка)

live-режим:  mic ──▶ [asr-stream] ──▶ [align-online] ──▶ позиция ──▶ [player]
```

Ключевой контракт всей системы — **`sync-map.json`**: список сегментов
`{ t_start, t_end, surah, ayah, word_range }`. Его одинаково производят и offline-конвейер,
и live-режим; его потребляет плеер. Пока есть этот контракт — модули развязаны.

## 2. Модули

### M1. `quran` — канонический текст (данные) ✅ ГОТОВО
Реализовано в `quran.py` (+ `test_quran.py`, 11/11). Чистый модуль без внешнего I/O, легко тестируется.
- Загрузка `quran.db` (схема: `surahs`, `surah_verses` с харакатами). Источник БД:
  `RisAbd/quran-data` (см. NOTES).
- Нормализация: снятие харакатов, приведение алифов/хамз, токенизация по словам.
  (Переиспользовать `text_without_hareke` / `ARABIC_LETTERS` из старого проекта.)
- Единый «поисковый корпус»: непрерывный поток нормализованных слов с обратным маппингом
  `индекс_слова → (surah, ayah, слово_в_аяте)`. Это фундамент для align.
- API: `load()`, `normalize(text)`, `tokens()`, `locate(word_index) → (surah, ayah, pos)`.

### M2. `ingest` — получение аудио (источник) ✅ (файл; YouTube — анти-бот)
- Вход: URL YouTube **или** путь к файлу.
- YouTube → `yt-dlp` (только аудио). Файл → как есть.
- `ffmpeg` → нормализация в 16 kHz mono WAV.
- Выход: `audio.wav` + `meta.json` (длительность, **исходная ссылка/источник**, исходное имя).
- **Хранить оригинальную ссылку** (для YouTube — video id): плеер показывает исходник рядом
  (см. правку П1 в §5). Основной кейс — ссылки на YouTube, можно на старте работать только с ними.
- Независим полностью.

### M3. `asr` — транскрипция (речь → текст+тайминги) ✅
- Вход: `audio.wav`. Выход: `transcript.json` — список `{word, start, end, confidence}`.
- Бэкенд: **локальный faster-whisper** (`large-v3`, GPU, float16 — уже проверено рабочим,
  см. NOTES про cuDNN). Отказ от Google Cloud STT старого проекта → убираем облако/креды.
- **Важно:** whisper по кораническому арабскому неидеален. Нам не нужна точная транскрипция —
  нужно «достаточно, чтобы найти место». Бэкенд за интерфейсом, чтобы позже подменить на
  специализированную модель (напр. Tarteel-подобную).
- Whisper даёт тайминги по сегментам; для пословных таймингов — `word_timestamps=True`
  (faster-whisper поддерживает) либо отдельный forced-alignment. Решить на этапе M3.

### M4. `align` — ЯДРО: привязка транскрипции к оригиналу ✅ v1 (работает, включая смену суры)
Самая сложная и новая часть. Именно здесь застряли в прошлый раз.
- Вход: `transcript.json` (M3) + корпус (M1). Выход: `sync-map.json`.
- Задача: сопоставить шумный поток ASR-слов с каноническим текстом и получить для каждого
  отрезка времени точное место `(surah, ayah, word_range)` + confidence.
- Подзадачи по возрастанию сложности:
  - **A. Известное место.** Пользователь сам указал суру/аят → выравниваем только внутри
    него по таймстемпам. «Относительно легко» (по словам автора). Первый рабочий путь.
  - **B. Авто-поиск в одной суре.** Найти место без подсказки (fuzzy sequence alignment:
    Needleman-Wunsch / `minineedle` уже в зависимостях, либо `rapidfuzz`; difflib из старого
    кода — слаб, апгрейдим).
  - **C. Переходы/смены места.** Тарауих: имам дочитывает суру и переключается на другую;
    в намазе смена суры разрешена. **Обязательно** детектить скачки позиции.
  - **D. Не-Коран/вставки.** Такбир («Аллаху акбар»), дуа, границы Фатихи — детектить и
    помечать/пропускать; опция «перемотать мимо».
- Разрабатывается **офлайн** на уже имеющихся сэмплах (`../speech-to-text-python/gcloud-speech-data/*/gstt_response.json`)
  без повторного прогона ASR.

### M5. `player` — UI (плеер с авто-прокруткой) ✅ v1 · 🔄 фокус-режим (П3/П4)
- Вход: `audio` + `sync-map.json` + текст (M1).
- Старт: **одна самодостаточная HTML-страница** (`<audio>` + JS), без сборки, без PyQt.
  Открывается и на телефоне. Крупный текст, подсветка текущего аята/слова, авто-скролл по
  таймстемпам.
- Навигация как chapters: прыжок на суру/место (для файла/ссылки).
- **Фокус-режим (П3):** в центре внимания — **текущий аят**, крупно; остальное приглушено
  (контекст: пара соседних строк). Не «весь структурированный документ», а поток с фокусом.
  Крупнее и «меньше за раз». Заголовки суры — лёгкие маркеры смены, не оглавление.
- **Индикатор позиции (П3):** маленький постоянный бейдж «где сейчас» (сура:аят), виден всегда.
- **Mobile-first (П4):** основной кейс — телефон; вёрстка и размеры сначала под мобилу.
- **Показ источника рядом (П1):** для YouTube — встроенное видео рядом с текстом (мини-окно
  или 50/50); текст крутится синхронно. (эмбед плеера YouTube ≠ скачивание — обходит анти-бот.)
- Позже — оптимизация/натив, если понадобится.
- Реализации: `web/player.html` (веб-фронт, актуальная) и `player.py` (`_HTML`, standalone).
  ⚠️ шаблоны разъезжаются — фокус-режим пока в `web/player.html`; синхронизировать позже.

### M6. `live` — живой режим (фаза 4)
- mic → streaming ASR (M3 в потоковом режиме) → `align` в инкрементальном режиме (M4) →
  текущая позиция → плеер обновляется на лету.
- Chapters/прыжков нет (поток живой). Основная тяжесть — онлайн-детект смен места (M4.C).

### M7. `service` — библиотека/очередь с обработкой (П2) 🆕 план · **Django + docker-compose**
Сейчас обработка ручная (`run.py` + `build_web.py`). Нужен сервис, куда **в любой момент
кидаешь любую ссылку**, а он берёт её в работу и показывает статус.
- **Add-by-link прямо с фронта:** поле «вставь ссылку» → создаётся запись.
- **Фоновая обработка:** очередь заданий (ingest → asr → align → forced-align → build). Ссылка
  сразу видна в списке со статусом `в очереди / обрабатывается / готово / ошибка`; «ждите».
- **CRUD + статусы:** список записей, добавить/удалить, кликнуть готовую → плеер; прогресс у
  обрабатываемых.
- **Стек (указка владельца): стандартная Django + docker-compose, всё конвенционально.**
  - `docker-compose`: web (Django/gunicorn) + db (Postgres) + worker (Celery/Django-Q) + брокер.
  - Модель `Recitation`: source_url, status, reciter, титулы, ссылка на sync-map/данные плеера.
  - API/вьюхи: add-by-link, list, status, delete; отдача плеера и аудио (Range).
  - Конвейер из `src/` вызывается воркером как **импортируемые модули** — ядро остаётся тонким,
    Django только оркестрация/хранилище/API. Тяжёлый ML (torch/wav2vec2) — в образе воркера
    (GPU-доступ) либо препроцесс на хосте (решить при реализации).
  - Фронт `web/` — под Django templates/static или как отдельный SPA к API.

## 3. Фазы (порядок работы)

- **Фаза 0 — каркас. ✅** Структура репо, забор `quran.db`, окружение/зависимости, контракты
  (JSON-схемы) `transcript.json` и `sync-map.json`.
- **Фаза 1 — MVP end-to-end. ✅** (даже без подсказки места — align сам находит) `ingest` + `asr` + `align.A` (место указано) +
  `player`. Цель — доказать сквозной путь и, главное, UX плеера в машине (то, что реально нужно).
- **Фаза 2 — авто-поиск.** `align.B`: находит место в одной суре без подсказки.
- **Фаза 3 — переходы.** `align.C/D`: мульти-сура/тарауих, детект скачков, вставки, chapters.
- **Фаза 4 — live.** `M6`.

Каждая фаза даёт работающий артефакт; модули между фазами не переписываются, а достраиваются.

## 4. Открытые вопросы

- Пословные тайминги: `word_timestamps` whisper vs отдельный forced-alignment — что точнее/легче.
- Метрика/порог уверенности для align (когда «нашли место» vs «не уверены»).
- Как детектить смену суры устойчиво (резкий разрыв в alignment-score? отдельный классификатор
  «Фатиха/такбир»?).
- Формат хранения готовых sync-map для «любимых» видео (кэш, чтобы в машине не считать заново).
- Качество арабского ASR у whisper на коранической рецитации — снять на реальных сэмплах,
  решить, нужна ли спец-модель уже в фазе 2.

## 5. Правки после демо (голосовое 2026-07-04, `media/audio_2026-07-04_01-15-09.ogg`)

Автор посмотрел рабочий прототип и надиктовал 4 пункта:

- **П1. Показывать источник рядом.** Хранить исходную ссылку; для YouTube — встроить видео
  рядом с текстом (мини-окно / 50/50), текст крутится синхронно. Чаще всего это ссылки на
  YouTube — можно пока только с ними. → M2 (хранить ссылку) + M5 (эмбед).
- **П2. Сервис с очередью и статусами.** Кидаешь любую ссылку в любой момент → берётся в
  обработку, показывает «в процессе». Полный CRUD, статусы (в очереди/обработка/готово),
  кликабельный список. → новый модуль **M7**.
- **П3. Фокус-режим отображения.** Текущий аят — крупно и в фокусе; контекст приглушён и
  необязателен; не весь структурированный документ. Крупнее и меньше за раз. Постоянный
  маленький индикатор «где сейчас». → M5. **✅ сделано в `web/player.html`.**
- **П4. Mobile-first.** Основная польза — на телефоне; тестировать на нём. → M5. **✅ вёрстка.**
- **П5. Пословная подсветка + плавный скролл за словом; амбиция — до хараке.** Не аят целиком, а
  текущее **слово** в фокусе, плеер плавно едет за словом. → M3/M4/M5. **Главный технический
  вызов** (здесь застряли раньше: whisper даёт слово «точкой», чтец тянет/паузит → тайминги
  разъезжаются). Решение — **forced alignment по известному тексту** (align.py локализует место,
  тайминги даёт CTC-аллайнер). Подробно — [`RESEARCH-word-alignment.md`](./RESEARCH-word-alignment.md).

Статус правок: **П3, П4 — сделаны** (веб-плеер). **П5 — в работе (spike).** **П1, П2 — в плане.**
