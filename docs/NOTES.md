## Идея владельца 10.07: forced (MMS) — сам себе распознаватель (целевая архитектура, (A))

Владелец (верно) заметил: акустическая модель внутри forced (MMS-CTC) выдаёт буквы/фонемы по
кадрам. Значит можно **не гонять google/whisper вовсе**:
1. greedy/beam CTC-декод эмиссий MMS → черновой фонетический текст;
2. сматчить на корпус Корана → сам определить читаемый отрывок (диапазон аятов);
3. затем forced-выравнивание известного текста.
→ Убирает зависимость от google (квота/креды) и от хлипкого `align.py` (кривые матчи слов, ложные
ранние якоря). Это «свой самодостаточный распознаватель» на одной модели. **Оговорка:** решает
«как найти отрывок», НЕ повторы (повторы = П8, ортогонально). Романизация MMS теряет харакат →
матчинг по скелету согласных, как в align.py, но без второго распознавателя. Средняя по объёму
работа; после П8. Записано как целевая архитектура.

## Ключевые решения

1. **ASR: локальный faster-whisper** вместо Google Cloud STT. Уже проверено рабочим на этой машине.
2. **Показываем оригинал, не транскрипцию.** ASR — только для поиска места и таймингов.
3. **Контракт `sync-map.json`** развязывает offline и live от плеера.
4. **Плеер — простая HTML-страница** (без PyQt), работает и на телефоне.
5. Разработку ядра (M4) ведём на существующих `gstt_response.json` — не жжём время на ASR в цикле.

## Технические заметки по транскрипции (fix cuDNN)

faster-whisper на GPU падал: `Unable to load libcudnn_ops.so.9` (exit 134). Лечится подстановкой
pip-путей к cuDNN/cuBLAS перед запуском:

```bash
export LD_LIBRARY_PATH="$HOME/.local/lib/python3.12/site-packages/nvidia/cudnn/lib:$HOME/.local/lib/python3.12/site-packages/nvidia/cublas/lib:$LD_LIBRARY_PATH"
```

`large-v3` float16 влезает в 6 ГБ RTX 3060; ~9 мин аудио считается меньше минуты.
(Сохранено также в личный skill `audio-task`.)

## Сервис: несколько распознавателей + хранилище по записям (2026-07)

Указки владельца: (1) завести виды распознавателей (whisper/google), гонять несколько и давать
выбор — сравнивать точность; (2) хранить сырые аутпуты рядом с аудио, раскладывать по папкам.

Архитектура:
- Модель **`AsrRun`** (FK на `Recitation`) = один прогон конвейера конкретным распознавателем
  поверх ОБЩЕГО аудио записи. Несколько прогонов = сравнение. `metrics` (coverage, wt, tl,
  duration, elapsed_sec) + `data` (выход build_data для плеера). `unique(recitation, recognizer)`.
- Реестр **`recitations/recognizers.py`** — `REGISTRY`/`PRIORITY`. Добавить распознаватель =
  запись тут + ветка в `pipeline._recognize`. Приоритет авто-выбора: google > whisper.
- Раскладка **`media/rec/<id>/`**: `audio.mp3` + `asr/<recognizer>/{raw.json,transcript.json,sync-map.json}`.
  `raw.json` — сырой ответ whisper/API как есть (дебаг). ingest — один раз на запись (`ensure_audio`).
- Пайплайн: `ensure_audio` (идемпотентно) → `run_one(run)` (asr→align→build) на каждый прогон.
  `tasks.run_pipeline` (вся запись) / `run_single` (один распознаватель, кнопка пересчёта).
- Плеер: `?asr=<key>` выбирает прогон (иначе авто); переключатель прогонов с метриками; статус
  записи агрегируется из прогонов. Legacy `Recitation.data` — фолбэк (миграция 0003 перенесла
  старые записи в прогоны: rec1→google, whisper-записи→whisper).
- google: raw.json прогона → кэш старого проекта (`gstt_key`/stem) → **живой Google STT API**
  (`src/gstt.py`). Живой путь: ffmpeg→FLAC 16k mono → blob бакета `SYNC_GSTT_BUCKET` →
  `long_running_recognize` (`ar`, без спец-модели — не поддержана) → формат `gstt_response.json`.
  Ключ ТОЛЬКО через env `GOOGLE_APPLICATION_CREDENTIALS` (в репо не коммитим, `.gitignore`),
  выключатель `SYNC_GSTT_LIVE=0`. Идемпотентно (raw.json/кэш) — пересчёт не жжёт квоту.
  Проверено: rec5 живым google coverage **1.0** vs whisper 0.722.
