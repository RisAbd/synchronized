## 1. Инвариант чтеца (фундаментальное правило)

Чтец во времени идёт по тексту так и только так:

- **вперёд ровно на одно слово** (нормальное чтение), ЛИБО
- **назад — возврат/перечитка** на любое расстояние (одно слово, несколько, целый аят и дальше;
  чтецы так делают постоянно), ЛИБО
- **держание** текущего слова (мадд — долгая гласная; пауза; вдох).

**Резкого прыжка ВПЕРЁД через слова (пропуск) физически не бывает.** Если подсветка перепрыгнула
через непрочитанные слова — это баг привязки, всегда.

Из инварианта следуют ДВЕ независимые оси отказа (важно их не путать — раньше путали):

| Ось | Что нарушено | Ловит | Проявление у владельца |
|-----|--------------|-------|------------------------|
| **A. Последовательность** | порядок слов по времени: сосед по времени прыгнул на +2 и дальше по тексту | `alignment_invariants` → `forward_jumps` | подсветка скачет вперёд через слова |
| **B. Привязка по времени** | слово стоит там, где акустика его НЕ подтверждает (время врёт, хотя порядок цел) | `acoustic_health` (§3.2) | «текст уезжает вперёд», подсветка обгоняет голос, мелькание, залипание |

Монотонный выравниватель (forced/w2v) по построению НЕ нарушает ось A внутри своего прохода, но
**легко нарушает ось B**: когда чтец возвращается/повторяет, а текст ему не дали переигрывать,
слова прижимаются друг к другу или растекаются по тишине — порядок цел (fj=0), а время врёт.
**fj=0 НЕ значит «привязка верная».** Это и есть корень «каши»: чинили ось A, не замечая ось B.

## 2. Термины

- **forced** — MMS-CTC выравниватель (`src/falign.py`), романизует арабский, роняет мадд → ложные дыры,
  но хорошо детектит возвраты по эмиссиям.
- **w2v** — wav2vec2/whisperx (`src/w2v_align.py`), держит мадд → честный coverage; монотонен, возвраты
  сам не выражает (наследует из forced, см. кейс D).
- **возврат (rep)** — чтец вернулся назад и перечитал; точки `rep=True` в forced sync-map.
- **зона возврата** — непрерывный диапазон слов в порядке чтения, чью раскладку сдвинул возврат.
- **эмиссии** — выход MMS перед CTC (`work/rec<id>_emissions.npy`); можно декодить greedy-CTC
  (`falign._greedy_ctc`) → романизованные буквы → сверять со скелетом слова (`falign._skeleton(
  falign._uroman_word(w))`). **Главный инструмент диагноза БЕЗ звука** (владелец: «глянь буквы,
  которые слышатся»).

## 3. Автоматические проверки (asserts на КАЖДОМ прогоне)

Владелец (голос 18.07): проверки — алгоритмические, в коде, автоматом на каждом прогоне, не ручные.
Кладутся в `run.metrics`, видны в листе/аудите.

### 3.1. `alignment_invariants(data)` — ось A ✅ реализовано

`service/recitations/pipeline.py`. Плоский индекс слова по канону (`_flat_index`): соседи по времени
должны идти +1 или ≤0, не ≥+2. Пишет `forward_jumps`, `collapsed_words` (слово мелькает <20мс),
детали. Аудит по всем прогонам: `tools/audit_alignment.py`.

### 3.2. `acoustic_health` — ось B 🔴 НЕ реализовано (нужен НАДЁЖНЫЙ сигнал, см. ниже)

Задумка: пометить слова, чья привязка по времени НЕ подтверждается акустикой (стоят в тишине /
не там, где реально читаются), — доля идёт в metrics как объективный индикатор дрейфа при fj=0.

**Проверено 2 наивных подхода — ОБА слишком шумные, чтобы на них гейтить (24.07, прототипы):**

1. **Скелетная сверка** (greedy-CTC декод окна `[t,t_next)` vs `_skeleton(_uroman_word(w))`, `_sim≥0.3`).
   Даже здоровые прогоны rec7 дают health≈0.44–0.56; google (cov 0.851) — худший 0.30. Короткие
   романизованные слова MMS декодятся слишком грязно → нет чистого порога «хорошо/плохо».
2. **RMS-тишина** (доля слов, чей онсет `[t,t+0.3с)` ниже порога речи = floor⁺¹⁵퍼 centile+10dB).
   Порог шумового пола НЕ сравним между записями: rec5 (здоровая, но тихий фон/паузы) → speech
   лишь 22% кадров → drift 0.43; rec10 → 2% кадров → drift 0.99. Медленное чтение с паузами
   неотличимо от дрейфа по абсолютному порогу. (Та же причина, по которой VAD отключён в ASR —
   на разнородном YouTube-таджвиде сегментация речь/тишина сама по себе ненадёжна.)

**Вывод:** абсолютной пороговой метрики оси B, сравнимой между записями, из этих сигналов НЕ выходит.
Нужен ОТНОСИТЕЛЬНЫЙ, привязанный к акустике САМОЙ записи сигнал. Кандидаты (не проверены):
- сравнивать онсеты прогона с **онсетами forced (MMS)** той же записи — forced даёт акустически
  обоснованные позиции слов; расхождение прогон↔forced ТАМ, где forced в здоровой зоне, = дрейф
  прогона (но forced сам дрейфует на кейсе E — нужен фильтр здоровья зоны, как в `_detect_repeats`);
- эмиссионная (не RMS) тишина — max-softmax эмиссий MMS в окне (низкая уверенность = нет речи),
  устойчивее к фоновому шуму, чем RMS (не проверено).

**Пока метрики нет — ось B диагностируем ВРУЧНУЮ по процессу §5** (декод эмиссий окнами). Не шипить
хрупкую эвристику: ложные тревоги на здоровых записях = та же «каша», от которой уходим.

## 4. Каталог кейсов

Каждый: **симптом → акустическая сигнатура → правило → текущая обработка → тест (запись+таймкод) → статус.**

### Кейс A — нормальное чтение вперёд ✅
- **Симптом:** норма, подсветка едет по одному слову.
- **Правило:** сосед по времени = +1 по индексу.
- **Обработка:** любой выравниватель.
- **Статус:** ок (эталон инварианта).

### Кейс B — одиночный возврат внутри аята ✅
- **Симптом:** чтец перечитал слово/несколько в том же аяте.
- **Акустика:** дыра в монотонном таймлайне + речь (RMS) + greedy-CTC-декод эмиссий совпадает со
  «назадними» словами эталона лучше, чем с «вперёдними».
- **Правило:** вклеить назадние точки (`rep=True`), фронт не трогать (bsearch по t).
- **Обработка:** `falign._detect_repeats` (Левенштейн ≥`_REPEAT_MIN_SIM`, tandem-collapse от удвоенного
  декода, добор перечитки `rb=max(rb,i)`).
- **Тест:** rec7 خضرا 6:99, ~1:28 (wi13→wi11→wi12→wi14). **Статус:** ок.

### Кейс C — держание на мадде (долгая гласная) ✅
- **Симптом:** долгий مادّ, forced роняет слово → дыра → залипание/скомканность.
- **Акустика:** одна held-гласная тянется секунды, согласных нет.
- **Правило:** не отсеивать stuffed-слова, а растянуть скомканный прогон по дыре; пред. слово держит
  фокус сквозь мадд.
- **Обработка:** `falign._stuffed_runs` + `_respace_stuffed` (опт-аут `SYNC_FALIGN_RESPACE=0`).
- **Тест:** rec11 Ан-Наджм 53:3→4 (الْهَوَىٰ держится ~4с). **Статус:** ок.

### Кейс D — кросс-аятный возврат (перечитка через границы аятов) ✅ закрыто 24.07
- **Симптом (w2v):** чтец вернулся и перечитал через границу аята; монотонный w2v затолкал слова
  СЛЕДУЮЩЕГО аята раньше по времени, чем перечитанные слова предыдущего → зигзаг по времени (ось A
  на w2v: fj>0).
- **Акустика:** зона расхождения forced↔w2v по времени (>1с) непрерывна и пересекает границу аята.
- **Правило:** наследуя возвраты из forced на w2v, замещать forced-точками ВСЮ непрерывную зону
  возврата в порядке чтения — **через границы аятов**, — а не по-аятно.
- **Обработка:** `pipeline._inherit_repeats` переписан на **flat-кластеры**: rep∪diverge-слова
  собираются в плоском порядке чтения, бьются на непрерывные кластеры (разрыв позиции >1 разделяет
  разные возвраты — честный w2v между ними не трогаем), каждый кластер `[lo..hi]` замещается forced.
- **Тест:** rec11 Ан-Наджм 53:53→(возврат 53:53)→54→55. Было w2v fj=3 (443→446→456 зигзаг),
  стало fj=0, время монотонно, возврат 53:53 выражен, cov=0.774. **Статус:** закрыто.
  (Раньше обходили дефолтом на forced; теперь w2v сам чист и выбирается по `min forward_jumps`.)

### Кейс E — похожие/повторяющиеся аяты + тихие провалы в хвосте → дрейф по времени 🔴 ОТКРЫТО
- **Симптом (владелец, голос 24.07, rec7 хвост после 2:00):** «текст уезжает вперёд, чтец с
  повторениями читает, похожие аяты идут, распознавание чуть-чуть вперёд бежит». Подсветка обгоняет
  голос, проносится через аят в почти-тишине.
- **Акустика (декод rec7 хвоста, эмиссии `work/rec7_emissions.npy`):**
  - 118–186с — сплошная речь, декод содержательный.
  - **186–204с — почти пусто** (нет чётких согласных): тихий провал/пауза чтеца.
  - w2v при этом расставил аят **102:0–15 на 188–202с** — прямо в тихом провале → подсветка
    проносится через весь 102, пока чтец молчит/читает не там.
  - 204–234 речь возобновляется (103), 234–242 снова провал, 242–256 речь.
  - Похожие аяты: 101 и 102 оба про «بديع السماوات والأرض / خلق كل شيء» → ASR/выравниватель путают,
    какой именно аят звучит.
- **Почему не ловится:** `forward_jumps=0` на всех прогонах rec7 — порядок слов цел. Это **ось B**
  (привязка по времени), а не A. Текущих автопроверок оси B НЕТ.
- **Первопричина (гипотеза):** монотонный выравниватель обязан разложить ВЕСЬ данный текст (аяты
  95–103) по всей длительности. Если чтец в хвосте повторяет/делает длинные паузы, а лишнего текста
  переигрывать нельзя, слова растекаются по тишине и прижимаются — время врёт. Возможно также, что
  диапазон аятов от ASR шире реально прочитанного (ASR «убежал» на похожем аяте) — тогда хвост текста
  вообще лишний.
- **Правило (проект спеки, к реализации):**
  1. Ось B должна проверяться автоматом (§3.2 `acoustic_health`) — слово, чьё окно акустически пустое
     или не бьётся со скелетом, помечается как недостоверная привязка; их доля идёт в metrics.
  2. Слова, попавшие в акустически пустую зону (тишина по RMS/эмиссиям), НЕ должны получать
     отдельные быстрые тайминги (подсветка не должна «бежать» по тишине) — держать предыдущее
     достоверное слово до появления речи, как в кейсе C держим на мадде.
  3. Если хвост текста лишний (ASR-диапазон шире прочитанного) — не растягивать его по тишине;
     проверить границу реально прочитанного по акустике.
- **Обработка:** пока нет. **Не замазывать заполнением** (пост-гард `fill_forward_jumps` был замазкой,
  откачен 18.07 — рисовал, будто пропущенные слова читаются; не воскрешать).
- **Тест:** rec7 Аль-Анам 6:95–103, хвост 186–204с (аят 102 в тишине), 234–242с. **Статус:** открыто.

### Кейс F — суры-рефрены (повтор одинаковых аятов) 🟡
- **Симптом (w2v):** сура с рефреном (Ар-Рахман «فبأي آلاء ربكما تكذبان»), w2v матчит рефрен на
  НЕ ту его копию → низкий coverage, битые t_end (t_end≤t).
- **Правило:** при равных строках привязывать к правильному вхождению по времени/порядку, не к
  первому попавшемуся; невалидные t_end (≤t) удалять.
- **Обработка:** частично — в `w2v_align.py` удаляем t_end≤t; выбор вхождения не решён.
- **Тест:** rec10 Ар-Рахман, cov≈0.626. **Статус:** частично.

### Кейс G — межсурные/межаятные прыжки ASR (не выравнивание) 🟡
- **Симптом:** google/whisper распознают НЕ тот диапазон аятов (убежали на похожем/следующем), из-за
  чего исходный диапазон для forced/w2v кривой.
- **Правило:** это дефект ASR/диапазона, а не выравнивателя; лечится на уровне выбора диапазона, не
  в falign.
- **Статус:** отдельная линия, не в П8.

## 5. Процесс (как пользоваться спекой на каждом баге)

1. **Вытащи случай:** запись + таймкод от владельца → `docker compose exec -T worker python
   manage.py shell` → достань forced/w2v word_timeline в зоне, посчитай `alignment_invariants`.
2. **Определи ось:** fj>0 в зоне → ось A (кейсы B/D). fj=0, но «текст уезжает/обгоняет» → ось B (кейс E).
3. **Декодь акустику** (без звука): `work/rec<id>_emissions.npy` (или посчитай заново),
   `falign._greedy_ctc` окнами → сверь со скелетами слов зоны. Смотри, что РЕАЛЬНО читается где.
4. **Сопоставь с каталогом §4.** Покрыт кейсом → чини по его правилу. Не покрыт → **добавь новый кейс**
   (симптом/акустика/правило/тест/статус) и правь код под правило.
5. **Проверь регрессии:** перегони/переприменённые прогоны, `tools/audit_alignment.py` — fj и (когда
   будет) acoustic_health по ВСЕМ записям; здоровые не должны деградировать.
6. **Обнови статус кейса** здесь + запись в BACKLOG/CLAUDE.md. Коммит+пуш.

**Грабли (не наступать):**
- Прерывание shell по таймауту оставляет осиротевший `gpu_align`, держит VRAM → CUDA OOM следующему.
  Чистить: `kill` gpu_align в контейнере (grep `/proc/*/cmdline`). w2v перегонять ПО ОДНОМУ.
- `build_data` (`src/player.py`) копирует только t/t_end/surah/ayah/wi — роняет `rep`/прочие флаги.
  Возврат в data выражается ДУБЛЕМ точки (то же слово дважды по времени), не флагом.
- Не воскрешать `fill_forward_jumps` (замазка).
