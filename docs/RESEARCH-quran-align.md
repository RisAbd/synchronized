## Что это

Инструмент пословной разметки **студийных пер-аятных записей** (формат EveryAyah:
`sssaaa.mp3`, один аят = один файл, текст аята известен заранее). На выходе — JSON с
сегментами `[word_start_idx, word_end_idx, start_msec, end_msec]` по каждому аяту
(индексация по словам `quran-uthmani.txt` Tanzil). В релизе 2016-11-24 — готовые разметки
**12 популярных чтецов** (Alafasy, Husary, Abdul Basit, Minshawy, Sudais, Shuraym и др.),
один zip 5.8MB. Качество (сверка с независимой реализацией ElMohafez): среднее расхождение
<73 мс, σ≈139 мс, сегментировано 98.5–99.9% слов.

## Пайплайн (4 стадии)

1. **Пер-чтецовая акустическая модель** CMU Sphinx, обучается на самих записях чтеца +
   коранической языковой модели. ⚠️ Скрипт подготовки обучающих данных **не опубликован**
   («in an unpublishable state») — т.е. **инструмент НЕ законченный/не turnkey**: без этой
   части воспроизвести стадию 1 нельзя.
2. **Распознавание PocketSphinx** по каждому аяту с **отфильтрованным словарём — только
   слова этого аята** (`segment.cc: ps_setup`). Резко поднимает точность/скорость: модель
   физически не может распознать «чужое» слово.
3. **DP-сопоставление распознанного с эталоном** (`match.cc`) — обычный
   Левенштейн/Нидлман-Вунш (штрафы 1/1/1) + разбор бэктрейса в **спаны**.
4. **Уточнение границ по аудио** (`segment.cc` + `discriminator.cc`): детект тишины,
   детект акустических переходов (мощность+MFCC), подтяжка границ слов.

Обвязка: `main.cc` — параллельные джобы по аятам, схлопывание мукатта'ат (в LM они
разбиты на буквы-«слова»); `align_blocks.py` — запуск блоками по 1000 аятов с ретраем
(в аллайнере/PS есть heap-corruption баг, автор не чинил — рестартует).

## Ключевые механики (по исходникам)

### match.cc — разбор DP-бэктрейса в спаны (главное наследуемое)

Вместо пословных догадок в мутных местах — **мульти-словные спаны** с общими границами:

- **Точное совпадение** ASR-слова с эталонным → закрыть текущий спан, отдельный спан из
  одного слова с таймингами ASR (`Exact`).
- **Неточное совпадение** (DP сматчил, текст различается) → копится в общий run-спан
  (`Inexact`), `stats.transpositions++`.
- **Слово эталона без ASR** (чтец «пропустил»/ASR не услышал) → run-спан, старт = конец
  предыдущего спана; спан закроется, когда синк восстановится. `stats.deletions++`.
- **Лишнее ASR-слово** (нет в эталоне — **сюда попадают и повторы чтеца!**) → слово НЕ
  добавляется, но **раздвигает временные границы** текущего спана. `stats.insertions++`.

То есть неуверенность честно выражена шириной спана, а счётчики ins/del/transp — готовая
метрика качества сопоставления.

### segment.cc — пост-обработка

- **Отсев невозможных спанов**: спан без ASR-опоры (`!MatchedInput`) короче
  **100 мс/слово** → выбросить («matcher stuffed a bunch of missing words into 10msec»).
  Это ровно наш симптом «скомканного начала»/фантомного аята (баг 3b) — у них решён
  ОБЩИМ правилом плотности, а не только по краям.
- **Тишина**: RMS-мощность окнами 50 мс, гистерезис −100 dbFS (вход) / −75 dbFS (выход)
  (`discriminate_silence_periods`). Дальше два правила:
  - старт слова не может лежать в тишине → сдвигается на конец тишины;
  - конец слова подтягивается к началу тишины перед следующим словом, иначе — вплотную
    к старту следующего (−10 мс). Убирает «слово размазано на паузу».
- **Liaise-точки** (внешний список: где слово кончается той же буквой, с которой
  начинается следующее — граница акустически неразличима): берётся ближайший
  акустический переход (пики скорости изменения мощности и MFCC-вектора, онлайн
  mean/σ с экспоненциальным затуханием, порог ~1.6–2.3σ; `discriminate_transitions_*`),
  и граница слов переносится туда (откат назад максимум 300 мс).

### Повторы чтеца (наш П8) — quran-align их НЕ решает

README прямо исключает повторы/пропуски из оценок точности («These results except…
where the qari repeated or skipped a phrase, <1% of all words»). В коде повтор — это
просто серия insertions: время повтора вливается в границы спана, подсветка «назад»
не возвращается. Для студийного мураттал-чтения по одному аяту это <1% и терпимо;
для наших живых читок (таджвид, частые возвраты — случай владельца) — нет.

**Но скелет правильный**: DP-сопоставление ASR↔эталон — та самая рамка, в которой
повторы выражаются естественно (запрещённый в классическом DP «ход назад по эталону»
надо просто разрешить с штрафом).

## Оценка: берём ли инструмент как есть? НЕТ

- Стек 2016 (CMU Sphinx/PocketSphinx) устарел; ключевой скрипт стадии 1 не опубликован;
  вход — пер-аятные студийные файлы, а у нас непрерывный YouTube-звук произвольных чтецов.
- Наш `falign.py` (MMS CTC) — современный и более сильный эквивалент стадий 1–2
  (не нужна пер-чтецовая модель вообще), а `align.py` (seed-and-consensus) уже решает
  то, чего у них нет в принципе, — поиск ЧТО читается в непрерывном аудио.
- Их gold-данные 12 чтецов — пригодятся отдельно (метрики ground truth, п.4 Next),
  файл: `quran-align-data-2016-11-24.zip` в Releases.

## Что НАСЛЕДУЕМ (конкретно, в порядке дешевизны)

1. **Правило плотности 100 мс/слово** → `falign.py`/`align.py`: любой участок, где k слов
   эталона уложены в < k×100 мс без опоры распознавания, — мусор, не отдавать в timeline.
   Обобщает наш краевой фикс 3b на ВНУТРЕННИЕ участки (дыры rec6 6.1с / rec7 12.1с из
   uxcheck — проверить этим же правилом обратную сторону: слова, съеденные интерполяцией).
2. **Подтяжка границ к тишине** (пост-шаг ПОСЛЕ выравнивания, НЕ фильтрация входа —
   грабли silero-VAD тут не применимы): RMS 50 мс + гистерезис −100/−75 dbFS, старт слова
   вне тишины, конец слова — к началу паузы. Дёшево (numpy по волне, без моделей). Лечит
   «перемотка мапит на прошлый аят» (гэпы приписываются последнему слову) и делает
   честнее `char_timeline`-интерполяцию (харакат-подсветка).
3. **Счётчики ins/del/transpositions против эталона диапазона** → в `metrics` прогона.
   Это конкретная форма идеи «сверка с ground-truth числом слов» из задачи про
   комплексную coverage-метрику (🟡 «Позже»): DP-сопоставить ASR-слова с текстом
   найденного диапазона аятов и репортить доли. Дешёвый, сравнимый между прогонами
   показатель «каши» в распознавании.
4. **П8 «возврат чтеца» — план реализации** (главный вывод для владельца):
   - Представление УЖЕ готово: `word_timeline` монотонен по времени `t`, а по
     позиции корпуса ходить назад МОЖЕТ; плеер ищет бинпоиском по `t` (uxcheck
     подтверждал) → **фронт менять не нужно, только аллайнер**.
   - Тактика: повтор = в DP-терминах серия «insertions», текст которых матчится на
     ТОЛЬКО ЧТО пройденное окно эталона. Пост-проход по сопоставлению: увидел run
     лишних ASR-слов → сматчить его на последние ~N слов эталона (n-грамма, с
     нормализацией как в `align.py`); совпало ≥2 слов подряд → эмитить точки timeline
     НАЗАД (повторное чтение), потом продолжить вперёд.
   - Где делать: НЕ в CTC-графе forced (фиксированный текст без повторов — сложно),
     а поверх плотного ASR (tarteel теперь даёт 876–992 слова на длинных читках —
     хватает). Схема: forced даёт точные границы линейного чтения; повторный проход
     по tarteel-словам добавляет обратные экскурсы. В `align.py` повторы коротких
     фраз сейчас глотаются консенсусом (окно WINDOW=4 голосует за доминирующую
     диагональ; run из 2–4 слов со сдвинутой диагональю проигрывает голосование либо
     режется MIN_SEG_WORDS=3) — потому и «залипает» подсветка (баг «چтец вернулся»).
   - Тест-кейс готов: rec6 (Аль-Ан'ам), `خَضِرًا` 1:24→1:27 и повтор 1:29→1:31
     (владелец дал точные тайминги), и «таких мест масса» дальше по записи.
5. **Позже (харакат-уровень, П5 стадия 2)**: детект акустических переходов
   (мощность+MFCC-скорость, онлайн-σ) для подтяжки границ там, где соседние слова
   сливаются (лиэйзон/идгам) — их liaise-механика с max_backtrack 300 мс.

## Не наследуем

- CMU Sphinx-стек, пер-чтецовые модели, пер-аятный вход — устарело/не наш кейс.
- `align_blocks.py` (рестарт при heap corruption) — лечение симптома, у нас нет.
- Схлопывание мукатта'ат — у нас текст из quran.db, токенизация уже наша.
